# 微信公众号爆款内容创作 Skill

## 这个 Skill 解决什么问题

帮助账号主在两个垂直领域（**AIGC 学习** + **审美想象力提升**）下，从选题、标题、行文、结构、发布到复盘，端到端地产出符合 2025-2026 年微信公众号算法推荐机制 + 人性传播规律的高流量内容。

当用户要求"做成 HTML"、"优化排版"、"粘到公众号"、"生成简约页面"时，默认输出一份**冷峻的程序员代码风格 HTML**：窄版正文、白色底、等宽编号、克制强调、无 JavaScript、无外链字体，便于预览、复制和二次修改。

当用户要求"配图"、"做首图"、"把占位图换掉"、"图文排版"时，默认沿用这套冷峻代码风。若用户要直接复制到微信公众号编辑器，优先使用纯 HTML/CSS 信息图区块，避免本地图片复制丢失；若只是本地预览或需要真实图片，再使用真实截图、方法图、清单图和项目内本地资产。

## 核心心法（务必先内化）

爆款 = **机制层** × **人性层**，两者缺一不可。

### 机制层（算法怎么放大）

公众号已不是纯订阅平台。流量主要来自三个池子：

- **看一看精选**（占比约 60%）：基于兴趣标签 + 社交关系
- **搜一搜**（约 15%）：依赖关键词密度和垂直度
- **文章底部推荐**（约 12%）：由完读率和互动数据驱动

算法把文章扔进一个 500-2000 人的初级流量池，用这群人的真实数据打分；分数过线进二级池，逐级跃迁。10万+ 的本质是连续通过 5-7 级跃迁。

判分核心指标（**这是创作时必须时刻挂在心上的几个数字**）：
- **完读率 ≥ 50%**
- **转发率 ≥ 8%**
- **48 小时互动率**（点赞 + 在看 + 评论 + 收藏）
- **关注转化率**
- **新号 30 天有"新手保护期"流量倾斜，必须用足**

### 人性层（人为什么愿意点开/读完/转发）

爆款文章至少踩中下列一项以上：
1. **情绪强度**（愤怒/焦虑/感动/共鸣/爽感/惊奇）
2. **社交货币**（转发能让转发者显得有品位/有见识/有思考）
3. **实用稀缺**（看完能用 + 别处没有）
4. **反共识**（用扎实论据颠覆主流认知）
5. **身份认同**（写给一个具体人群，让他们觉得"这说的就是我"）
6. **时机性**（嫁接热点到自己的垂直领域）

**爆款公式**：
```
爆款 = 垂直选题 × (情绪 + 社交货币) × 标题钩子 × 完读节奏 × 发布时机
```
任何一项是 0，结果就是 0。

## 标准工作流

### 工作流 A：选题研究（用户说"帮我想选题"/"这周写什么"/"找点 idea"）

1. **确认领域**：是 AIGC 还是审美？如果不明确就直接问一句"这次想做 AIGC 还是审美方向？"
2. **加载领域文件**：读取对应的 `references/domain-aigc.md` 或 `references/domain-aesthetic.md`，了解该领域的内容来源清单和读者画像
3. **加载选题方法**：读取 `references/topic-research.md`，按里面的"五维选题挖掘法"执行
4. **网页浏览研究**：用 `web_search` / `web_fetch` 按 `topic-research.md` 里指定的源头清单去扒最新热点和高赞内容
5. **打分与输出**：给出 8-15 个候选选题，每个选题用以下结构呈现：

```
选题：[一句话标题方向]
钩子角度：[情绪 / 社交货币 / 实用 / 反共识 中的哪一种]
预估读者：[具体人群画像]
信息源头：[扒到这个选题的来源链接]
传播指数：⭐⭐⭐⭐⭐（5星制，基于情绪强度+稀缺度+时效性综合判断）
建议标题方向：[2-3 个标题方向]
```

按传播指数倒序排列，最后用一句话推荐 Top 3。

### 工作流 B：写一篇完整文章（用户说"写一篇关于 X 的公众号文"/"帮我写爆款"）

1. **确认领域和选题**：领域是 AIGC 还是审美？选题是用户给定还是要从工作流 A 选一个？
2. **加载领域文件**：对应的 `domain-*.md`，掌握该领域的语言风格、读者痛点、用词偏好
3. **加载写作工艺**：读取 `references/writing-craft.md` + `assets/article-templates.md`
4. **先输出标题方案**：读取 `assets/title-formulas.md`，按里面的公式生成 **8 个不同方向的标题**，按"打开欲望"排序，让用户选或者你自己选最强的
5. **先输出开头 150 字**：开头是生死线。用 `writing-craft.md` 里的开头钩子模板写 2 个版本给用户挑
6. **输出完整正文**：按选定的文章结构模板展开，严格遵守 `writing-craft.md` 的节奏规范（每 300-500 字一个钩子、关键信息加粗、避免"赋能/链路/抓手"等八股词、说人话）
7. **写转发钩子结尾**：一个金句 / 一个能让读者"代我立言"的观点
8. **质量自检**：用下面的 Pre-Flight Checklist 走一遍

### 工作流 C：优化已有文章（用户说"帮我改改这篇"/"为什么这篇没流量"）

1. **诊断方向**：先问数据——完读率多少？转发率多少？关注转化多少？据此定位问题：
   - 打开率低 → 标题/封面问题，重做 8 个标题候选
   - 完读率低 → 开头不抓人 + 节奏问题，加钩子、砍废话、加粗、拆段
   - 转发低 → 缺社交货币或情绪强度不够，重写结尾金句，加观点态度
   - 关注转化低 → 单篇内容和账号定位脱节，调整切入角度
2. **逐段改写**：不要整篇推翻，按问题精准改

### 工作流 D：数据复盘 / 长期增长策略（用户问运营策略）

加载 `references/platform-mechanics.md`，按里面的"复盘四问"框架引导用户分析数据，给出下周内容方向调整建议。

### 工作流 E：输出冷峻代码风 HTML（用户说"做成 HTML"/"排版优化"/"粘到公众号"）

1. **先定稿再排版**：如果正文还没完成，先按工作流 B/C 完成文章内容；不要一边写一边堆 HTML。
2. **加载排版规范**：读取 `references/html-layout.md`，再读取 `assets/template.html` 作为页面骨架。
3. **重组文章层级**：把内容整理成 `title`、`subtitle`、`meta`、`lead`、3-6 个 `section`、`closing`、`references`。没有真实来源时不要编来源。
4. **套用模板**：保留模板的白色背景、等宽编号、代码注释式 kicker、左侧细线、克制高亮和媒体占位框。不要换成杂志风、营销风、暖色卡片风。
5. **公众号友好**：禁止 JavaScript、外链字体、复杂动效、表单、iframe。图片使用占位框或用户提供的本地/网络图；正文不要依赖 hover 才能理解。
6. **保存文件**：在当前工作目录保存为语义化文件名，如 `wechat-article-aigc-sidejob.html`。如果用户指定文件名，以用户指定为准。
7. **交付前自检**：检查标题已同步到 `<title>` 和页面 H1；移动端宽度不溢出；全文没有 markdown 残留；HTML 能直接在浏览器打开。

### 工作流 F：文章配图（用户说"配图"/"做首图"/"把图片占位换掉"）

1. **先读配图规范**：读取 `references/image-layout.md` 和 `references/html-layout.md`，确认图片数量、风格、资产路径和公众号友好要求。
2. **判断图片类型**：首图 / 解释图 / 清单图 / 真实截图。AIGC 方法文优先用 SVG/HTML 方法图；需要照片感、插画感、复杂场景时再调用图像生成。
3. **选择复制策略**：如果用户要复制到公众号编辑器，优先用 `.html-visual`、`.visual-grid`、`.visual-panel` 等纯 HTML/CSS 区块；如果只是本地预览或需要真实截图，再建立 `article-assets/<article-slug>/` 并用相对路径引用图片。
4. **建立资产目录**：需要真实图片时，为每篇文章创建 `article-assets/<article-slug>/`，图片用语义化文件名，HTML 使用相对路径引用。
5. **替换占位图**：如果 HTML 里有 `.media-slot`，可先运行 `scripts/illustrate_html.py --plan` 生成配图计划，再用 `--config ... --apply` 批量替换；复制友好版本可以手动插入 `.html-visual` 区块。
6. **写清 alt / role 和 caption**：图片文件用 `alt`，HTML 信息图区块用 `role="img"` 和 `aria-label`，`caption` 说明图的用途或来源。真实截图、数据图、论文图不要伪造来源。
7. **浏览器检查**：打开或刷新 HTML，检查图片或信息图区块是否显示、移动端是否溢出、是否仍有 `IMAGE SLOT` 或模板说明残留。

### 工作流 G：公众号编辑器无缝中转（用户说"复制过去丢图"/"有没有别的通路"）

1. **先判断目标**：如果用户只是本地预览，继续交付 HTML；如果目标是复制到微信公众号编辑器，优先做 `.docx` 中转版。
2. **生成嵌入式图片**：把 HTML 信息图或 SVG 配图渲染成 PNG，再嵌入 `.docx`，不要只在文档里放本地图片路径。
3. **重排成 Word 友好结构**：标题、导语、正文小标题、引用、图片、caption、references 都用 Word 原生段落和内嵌图片。不要依赖网页 CSS。
4. **交付方法**：告诉用户打开 `.docx`，在 Word / WPS / Pages 中全选复制，再粘贴到公众号编辑器。图片会以剪贴板图片进入编辑器，成功率高于从浏览器复制 HTML。
5. **更高阶通路**：如果用户有公众号 AppID / Secret 或愿意打开官方编辑器，可以走 API 草稿创建或浏览器自动上传图片，但这需要账号权限和用户确认。

### 工作流 H：封面图生成（用户说"封面图"/"首图"/"题图"/"吸睛"）

1. **先读封面模板**：读取 `assets/cover-image-prompts.md` 和 `references/image-layout.md`，不要把文章内配图规则直接套到封面图上。
2. **提取封面变量**：从文章提取 `title`、`domain`、`emotion`、`visual_metaphor`、`cover_role`。标题必须逐字准确。
3. **选择有字 / 无字**：用户明确要“带标题”时生成有字封面；如果中文标题生成风险较高或用户要自己排字，生成无字封面并保留标题安全区。
4. **调用图像生成**：使用 imagegen 生成 16:9 横图。AIGC 默认浅色高级科技感，审美默认浅色编辑部质感。
5. **保存到项目目录**：把生成结果从默认生成目录复制到 `article-assets/<article-slug>/cover-title-<slug>.png`，不要只留下临时路径。
6. **检查可用性**：打开图片检查标题是否正确、缩略图是否清楚、没有水印/多余文字/廉价 AI 元素。标题错字必须重做。
7. **交付路径**：返回项目内图片路径；如果用户在做公众号，也建议把这张图作为后台封面上传，不放进正文复制链路。

## Pre-Flight Checklist（每篇发出前必须过一遍）

在交付任何文章前，逐项打勾：

- [ ] **垂直度**：这篇文章符合账号定位的 3-5 个核心标签吗？算法能一眼看懂这是什么内容吗？
- [ ] **前 150 字钩子**：开头三句话内有没有故事/反常识/具体场景把人留住？
- [ ] **标题三要素**：包含"具体数字/身份标签/反差/悬念/利益点"中至少两个？
- [ ] **非震惊体**：标题没有"震惊体"、"厉害了"、"解密 XXX" 等触发限流的词？
- [ ] **完读节奏**：每 300-500 字有一个小标题、加粗、问句或转折？
- [ ] **说人话**：通篇没有"赋能/链路/抓手/颗粒度/打法"这类八股词？
- [ ] **社交货币**：读完后读者愿意拿它当人设道具转发吗？
- [ ] **转发钩子结尾**：结尾有金句/观点/能让读者"代我立言"的句子吗？
- [ ] **领域风格匹配**：AIGC 是不是"说人话+实用感"？审美是不是"有质感+不装"？
- [ ] **合规**：没有夸大宣传、没有医疗金融政治敏感雷区？
- [ ] **HTML 可交付**：如果输出 HTML，是否使用 `assets/template.html`，且无 JS、无外链字体、无 markdown 残留、标题和正文层级正确？
- [ ] **配图可交付**：如果有配图，复制到公众号是否不会丢图；若用图片文件，是否在项目目录内、路径可打开、alt/caption 完整、没有无意义氛围图？

任何一项打不上勾，就回头改，不要直接交付。

## 与用户协作的语气

用户是公众号主，不是技术人员。沟通时：
- 用"我们这篇文章"而不是"该文章"
- 提选题/标题时直接给方案 + 推荐，不要让用户自己排列组合
- 引用方法论时说清楚"这是基于完读率/转发率的考虑"，让用户理解 why
- 如果用户给的素材太弱（信息密度低、没角度），直接说"这个素材撑不起一篇爆款，建议换 X 角度或加 Y 维度"，不要硬写

## 跨平台兼容性说明

这个 skill 的所有文件都是纯 markdown，没有任何平台特定的语法。无论是在 **Claude Code**、**OpenAI Codex CLI**，还是其他支持 markdown 加载的 agent 工具中，都按以下方式使用：

1. 将整个 `wechat-viral-content/` 文件夹放到 agent 能访问的工作目录或 skill 注册路径
2. 在对话中，agent 读取 `SKILL.md` 作为入口，按文中"何时加载哪个引用文件"的指引按需读取 `references/` 和 `assets/`
3. 如果 agent 没有 `web_search`/`web_fetch` 工具，跳过工作流 A 的步骤 4，改为让用户自己提供热点信息或贴文链接

## 反模式（不要做的事）

- ❌ 凭记忆给方法论。每次执行任务前都要读取对应的 reference 文件
- ❌ 一上来就写正文。**永远先做选题/标题/开头**这三个高 ROI 环节
- ❌ 标题用"震惊体"或夸张套路。2025 年算法识别准确率已 92%，会直接限流
- ❌ 写空泛的"赋能、链路、抓手、闭环"。这是公众号沉降市场后的反向毒药
- ❌ 给用户"10 个方向自己选"。要做判断、给推荐、说理由
- ❌ AIGC 文章里堆技术黑话，审美文章里堆文艺腔。两个领域都要"说人话"，但风味不同（详见 `domain-*.md`）
